This is the living engineering guide for Demo Co — a single, self-contained page that summarizes how we build, run, and reason about our systems. Architecture at a Glance

Demo Co runs a Node + TypeScript monorepo on one shared Postgres, deployed to Kubernetes on EKS. A single public gateway is the only authenticated edge in front of four internal backend services; the browser only ever talks to the gateway.

  • Three repos. (backend API), (frontend), and (infrastructure). Each repo is authoritative for its own domain; on conflict, the code wins for behavior.
  • One public edge. The gateway is the single public entrypoint and the only authenticated edge. Internal services are not reachable from outside the cluster.
  • Five backend services. gateway, accounts, billing, usage, notifications.
  • One shared Postgres, schema-per-service. Each service owns its own schema and tables; no service reads another service's tables.
  • No direct service-to-service calls. A cross-service read goes back out through the gateway over HTTP.

How a request flows

platform Service catalog (wiki article 1008)

Every external call enters through the gateway, which authenticates it before forwarding. The Gateway class wires this up. Session and cookie detail is owned by Authentication & Authorization.

  1. Browser calls the gateway. A request hits https://api.demo.co/<service>/... carrying the demo_session cookie.
  2. Gateway verifies the session. verifySession rejects anything without a valid, unexpired session.
  3. Gateway proxies inward. proxy forwards to the internal service, adding x-user-id and x-org-id from the verified session.
  4. The service does its work. Scoped to that org, it queries its own schema and returns JSON.

The gateway routes exactly four public prefixes, one per internal service:

  • /accounts
  • /billing
  • /usage
  • /notifications

The health probe is the one exception to auth. GET /healthz is registered before verifySession, so it does not require a session; everything else does.

Trust the header, never the bodyInternal services trust x-org-id as set by the gateway and must never accept an orgId from the request body. Every query is scoped to the caller's org.

Why this shape (the ADRs)

Architecture Decision Records (wiki article 1001)

Two decisions explain the whole map. The wiki (article 1001) is canonical for ADR numbering.

ADR-001 — API gateway as the single entrypoint
All external traffic goes through one gateway; internal services are not publicly reachable and never call each other. The rationale is one place for auth, rate limiting, and audit logging.
ADR-002 — Schema-per-service in a shared Postgres
Each service owns its tables within one shared Postgres instance. Shared Postgres was chosen over database-per-service to keep ops simple at the current scale.
Numbering noteThe platform doc docs/architecture.md swaps these two ADR numbers and uses different titles. The wiki is canonical for ADRs — follow the numbering above and do not trust the platform doc's numbers here.

Demo Co also practices trunk-based development, shipping from main. The exact ADR text for that practice is not in the available sources, so it is described as the practice only. Migrations live under platform/services/<name>/migrations, are applied by make migrate, and a service never migrates another's schema; see Database Migrations.

Per-service detail

Service catalog (wiki article 1008) platform

Gateway

The Gateway is an express app and the single public entrypoint. Its constructor wires three things in order:

  1. Health probe. /healthz with no auth — the liveness/readiness probe.
  2. Session check. verifySession gates everything else.
  3. Proxy. proxy forwards the public prefixes /accounts, /billing, /usage, and /notifications to the internal services, adding x-user-id and x-org-id from the verified session.

The gateway listens on port 3000, and internal services resolve at http://<name>.internal inside the cluster. The full request lifecycle is owned by The Gateway & Request Flow; networking, replicas, and container ports by Infrastructure & Environments.

Design intent vs. implementedThe single-entrypoint decision names the gateway as the one place for auth, rate limiting, and audit logging. The read service files implement auth, routing, and the health probe; rate limiting and audit logging are design intent here — see The Gateway & Request Flow.

Accounts

The AccountsService owns identity, orgs, membership, and roles. The role type is owner | admin | member, and the first user of an org is always its owner.

listMembers
Returns the members of an org.
createOrg
Creates an org; the creating user becomes its owner.
inviteMember
Owners/admins only; enqueues a member_invite notification through the gateway.
setRole
Changes a member's role; transferring ownership demotes the previous owner to admin, keeping exactly one owner per org.

Roles, sessions, and API keys are owned by Authentication & Authorization.

Billing

The BillingService owns plans, invoices, and payment. Invoice status is draft | open | paid | void and plan name is free | team | enterprise.

invoicesForOrg
Lists the invoices for an org.
finalizeInvoice
Charges base plan price plus overage, moves the invoice draft→open, and enqueues an invoice_receipt. It reads the usage total via GET /usage/total?period=current through the gateway — never the usage schema directly.
chargeInvoice
Moves open→paid on success; failed charges retry for 7 days.

Plans, overage math, and the invoice state machine are owned by Billing, Plans & Usage Metering.

Usage

The UsageService meters one row per metered action. The usage_events table is append-only — rows are never updated or deleted.

record
Append-only INSERT of units for an org.
totalForPeriod
Sums units in a time window; this is what billing reads through the gateway when finalizing an invoice.

Notifications

The NotificationsService sends transactional email and webhooks. Channels are email | webhook and templates are invoice_receipt | member_invite | usage_alert | payment_failed.

enqueue
Inserts a row with status pending; the channel defaults to email.
deliverPending
A background worker tick that claims pending rows with FOR UPDATE SKIP LOCKED, marks each sent or failed, and retries failures with backoff.

Channels, templates, and the delivery worker are owned by Notifications & Webhooks.

Core Domain & Data Model

The domain centers on the organization (org) — the top-level tenant that owns members, plans, invoices, usage, and notifications. Each service owns its own Postgres schema (Architecture at a Glance, ADR-002), so the tables below are grouped by the service that owns them, and this section is the canonical home for column-level schema detail.

In shortUUID primary keys (gen_random_uuid()); money as integer cents; timestamps as timestamptz in UTC. Every table except plans carries org_id. Only accounts.users → accounts.orgs is an enforced foreign key; other org_id links are logical and cross-schema.

Conventions

platform Architecture Decision Records (wiki article 1001)

These rules hold across every schema. They are not optional house style — the migrations enforce them and the TypeScript interfaces mirror them.

Schema-per-service
Every table lives in its owning service's schema inside one shared Postgres. No service reads another service's tables — see Platform Services for the ownership map.
UUID identifiers
Primary keys are uuid with default gen_random_uuid().
Money as integer cents
Amounts are stored as integer cents (e.g. amount_cents, monthly_price_cents) — never floats.
Timestamps
Time columns are timestamptz in UTC, commonly DEFAULT now().
Tenant scoping
Every table except plans carries org_id. Every service method is scoped to the org the gateway passes; a service must never trust an orgId from a request body. A missing org scope is a security bug — see Security & Secrets.

The Gateway & Request Flow

Every request enters through one door: the gateway. It is the only publicly-exposed service and the only authenticated edge — it verifies the session, attaches the caller's identity as headers, and forwards the request to an internal service that does the real work and returns JSON.

  • One entrypoint. The browser calls https://api.demo.co/<service>/...; internal services are never reachable directly.
  • Auth at the edge. The gateway runs verifySession and rejects anything without a valid, unexpired session with a 401.
  • Identity by header. The gateway forwards to http://<name>.internal, adding x-user-id and x-org-id from the verified session.
  • Trust the headers. Internal services scope every query to x-org-id and must never accept an orgId from the request body.
  • Cross-service reads loop back. Services never call each other directly — a read goes back out through the gateway.

The request lifecycle, end to end

platform Architecture Decision Records (wiki article 1001)

A single request travels through four steps. The gateway owns steps 1–3; the internal service owns step 4 and hands its result back through the gateway.

  1. Browser sends the cookie. The frontend calls https://api.demo.co/<service>/... carrying the session cookie demo_session. The api-client sends it with credentials: "include" — see Frontend Development.
  2. Gateway verifies the session. verifySession reads the opaque demo_session cookie and resolves it to identity. A missing token returns 401 {"error":"no session"}; an expired or unresolved session returns 401 {"error":"session expired"}. On success it stashes the resolved session (userId, orgId, expiresAt) on the request and continues.
  3. Gateway proxies inward. The proxy factory forwards the request to http://<name>.internal, preserving the method and adding identity headers x-user-id and x-org-id.
  4. Service responds. The internal service does its work scoped to that org, returns JSON, and the gateway relays the upstream status and body straight back to the caller.

Authentication & Authorization

Authentication answers who you are; authorization answers what you may do. At Demo Co both are settled at the gateway and in the services behind it: an opaque demo_session cookie identifies the caller, and a server-side role decides what that caller can do inside their org.

  • Sessions. The browser sends an opaque demo_session cookie; the gateway verifies it, checks expiry, and attaches identity headers before forwarding inward.
  • Roles. owner | admin | member, enforced server-side. The frontend hiding a control is convenience, not security.
  • One owner per org. Enforced by a unique partial index; transferring ownership demotes the previous owner to admin.
  • API keys. Bearer tokens (demo_sk_) with admin-level permissions, shown once, revocable instantly.
  • SSO/SAML. Enterprise plan only; first-time SSO sign-in provisions a member; keep a break-glass owner with a password.
Where this livesThis section owns auth, sessions, API keys, and SSO. The request lifecycle and header injection are owned by The Gateway & Request Flow; security policy by Security & Secrets; plan gating for SSO by Billing, Plans & Usage Metering.

Org-scoping: never trust a body orgId

This is the central backend convention. Every service method is scoped to an org, and the only trusted source of that org is the gateway. After the gateway verifies the session, its proxy attaches two identity headers from that session — x-user-id and x-org-id — and downstream services scope every query to the verified org.

  • Take orgId first. Method signatures lead with orgId so scoping is unmissable.
  • Trust the header, not the body. Scope queries to x-org-id; never read an orgId from the request body.
  • It is a security invariant. A missing org scope is a security bug — see Security & Secrets.
Never trust an orgId from a request bodyThe verified org comes from the session, relayed as x-org-id. Reading an org identifier the client could set would let one tenant read another's data.

Coding Standards & Conventions

A handful of conventions keep two repos and five services consistent. Store money as integer cents and ids as UUIDs; scope every backend query to the caller's org and never trust an orgId from the request body; reach the backend only through api-client and build UI only from ui; keep pull requests small and let the linter own style.

  • Data. UUID ids, money in integer cents, timestamps as timestamptz in UTC, inputs validated with zod.
  • Backend. Org-scope every query, ignore any body orgId, never read another service's schema.
  • Frontend. All backend calls go through api-client; all primitives come from ui.
  • Process. One idea per pull request; review for correctness and tests; let eslint handle style.

Backend conventions

Never trust a body orgId

No cross-service DB reads
A service never queries another service's schema. Cross-service data goes back out through the gateway over HTTP — billing reads usage totals from the usage service that way, never from the usage schema directly.
A missing org scope is a security bugTreat an unscoped query the same as a vulnerability, not a code-style nit. The gateway is the only thing that establishes identity; once a request is inside a service, the only trustworthy org is the one in x-org-id.

Database Migrations

Change the schema in two deploys: expand first, then contract. Demo Co ships continuously and Kubernetes rolls pods one at a time, so for a short window old and new code run against the same database — every migration must be safe for that overlap.

In shortExpand then contract over two deploys; each service owns its migrations under platform/services/<name>/migrations; make migrate applies them in filename order; add columns nullable or with a default, backfill in batches, build indexes CONCURRENTLY; never rename or drop a column in the same deploy that stops using it; a failed migration fails the deploy — fix forward.

This section owns migration mechanics. The deploy pipeline and rollout behavior live in Release & Deployment; the column-level schema lives in Core Domain & Data Model. All migration tooling is in .

Why expand/contract

Database migrations (wiki article 1007) runbooks/deploys.md

Migrations are applied as part of the release, before the new image takes traffic. Because pods roll one at a time, old pods keep serving against the new schema during the rollout, so every migration must be backward compatible with the code currently running.

The way to satisfy that is to split a breaking change across two deploys:

Expand (deploy 1)
Add the new shape alongside the old one. The schema now supports both the old and new code, so old pods keep working while the rollout proceeds.
Contract (deploy 2)
Once every pod runs the new code and nothing reads the old shape, remove it.

A worked rename: full_name → display_name

Database migrations (wiki article 1007)

This is the documented pattern, not a committed file — renaming a column directly would break old pods mid-rollout. Split it in two:

  2. Deploy 2 (contract). Stop writing full_name, then drop it. By now every pod reads display_name, so nothing depends on the old column.

Release & Deployment

Demo Co ships from main continuously. Every merge to main runs CI, builds container images, and deploys to staging automatically; production is a tagged release — push a git tag matching v* and the same pipeline deploys to production.

This section owns the deploy pipeline and rollback. The short version is below; the rest of the section is the detail.

  • Merge to staging is automatic. A push to main triggers the pipeline and lands on staging with no manual step.
  • Production is a tag. Cutting a v* git tag is the promotion; there is no separate "deploy to prod" button.
  • Soak first. Let a change sit on staging for roughly 30 minutes before tagging — this is a human judgement, not an enforced gate.
  • Three jobs, chained. test → build-and-push → deploy. A red build never deploys.
  • Rolling deploys. Kubernetes replaces pods one at a time; the gateway readiness probe gates traffic, so a bad image fails the rollout instead of taking the site down.
  • Fastest undo. kubectl rollout undo deploy/gateway, or re-tag the last good commit. Never hand-edit production.

Rolling deploys and the health gate

Kubernetes rolls pods one at a time. The readiness probe gates traffic to each new pod, so a bad image fails the rollout rather than taking the site down. The gateway runs replicas: 3, and its readiness probe is GET /healthz on port 3000.

  • The probe. The gateway implements GET /healthz, returning { ok: true }; the manifest checks it with an initialDelaySeconds of 5.
  • No auth on the probe. /healthz requires no session — that contract is owned by The Gateway & Request Flow.
  • Bad image, contained. If new pods never go ready, traffic stays on the old pods and the rollout stalls instead of erroring out to customers.

Security & Secrets

Security at Demo Co rests on two rules: the gateway is the only authenticated edge, and every query is scoped to the caller's org. Identity flows inward as x-user-id / x-org-id headers the gateway sets, and secrets live in AWS Secrets Manager — never in git.

  • One authenticated edge. The gateway verifies the session and sets identity headers; internal services trust those headers and never accept an orgId from a request body.
  • Org scoping is mandatory. Every query is scoped to the caller's org. A missing org scope is a security bug, not a nice-to-have.
  • Roles server-side. owner | admin | member are enforced on the server. The frontend hiding a button is convenience, not security.
  • Secrets in Secrets Manager. They reach pods as Kubernetes Secret env vars via secretKeyRef. Never commit a secret; rotate on a schedule and immediately on suspected exposure.
  • Report fast. Revoke first if you can, then report in #security. Sev 1 security issues follow the incident process.

The mechanics of sessions, API keys, and SSO are owned by Authentication & Authorization; the request lifecycle and identity-header detail by The Gateway & Request Flow; cloud, Kubernetes, and Secrets Manager plumbing by Infrastructure & Environments.

The authorization model

Security & secrets (wiki article 1009) platform

The gateway is the only authenticated edge (ADR-001). It verifies the session, then sets x-user-id and x-org-id from the verified identity before forwarding inward. Internal services are not reachable from outside the cluster, so they trust those headers — but only those headers.

The single most important rule for backend code: the caller's org comes from the header the gateway set, never from the request. Internal services must never accept an orgId from a request body; they scope every query to x-org-id and ignore any body orgId. See proxy in .

Identity is set, not asked
The gateway runs verifySession, then copies the verified identity onto x-user-id / x-org-id. Downstream services do not re-read the cookie.
Every query is org-scoped

Health check is the one exception
GET /healthz must not require auth; everything past it requires a valid session.
Never trust a body orgIdIf a handler reads the org from the request body instead of x-org-id, a caller can read or write another org's data. Trust the header the gateway set; ignore the body. Session lookup, expiry, and the 401 responses are detailed in Authentication & Authorization.

Product-facing keys & SSO

API keys SSO & SAML

Customers manage their own credentials, and the mechanics are owned by Authentication & Authorization. From a security standpoint the points that matter are the blast radius and the speed of revocation.

API keys
Created at Settings -> API keys (owners and admins only), shown in full once, and sent as a bearer token to https://api.demo.co (example prefix demo_sk_). A key has the same permissions as an admin, so store it in a secret manager — never source control.
Revocation
Revoke immediately if a key may have leaked. "Revocation takes effect at once and cannot be undone." Rotate by creating a new key, updating integrations, then deleting the old one.
SSO / SAML
Enterprise plan only; SAML 2.0 with Okta, Entra ID, or Google Workspace. Just-in-time provisioning adds a first-time SSO user as a member. When requiring SSO, keep one owner who can still sign in by password as a break-glass account.

Reporting a security issue

Security & secrets (wiki article 1009) Incident management (wiki article 1006)

If something looks wrong, act before you write it up. "Revoke first if you can, then report in #security." Every report is triaged, and Sev 1 security issues follow the incident process.

  1. Contain. Revoke the exposed credential or key first if you are able — revocation is immediate.
  2. Report. Post in #security with what happened and what you have already done.
  3. Escalate if severe. A Sev 1 security issue (full outage or data-loss risk) runs through Incident Response & Postmortems — declare, mitigate, communicate, resolve, postmortem.
When in doubt, ask"When in doubt, ask in #security." Over-reporting a non-issue is cheap; sitting on a real one is not.

ADR (Architecture Decision Record)
A short record of a significant architecture decision; the wiki is canonical for numbering. ADR-001 is "API gateway as the single entrypoint" and ADR-002 is "Schema-per-service in a shared Postgres" (wiki article 1001). The platform doc swaps these numbers — follow the wiki.
admin
The middle role (owner | admin | member); can manage members, create API keys, and the like. Roles are enforced server-side — see Authentication & Authorization.
API key
A bearer token (prefix demo_sk_) sent to https://api.demo.co, created at Settings → API keys by owners or admins only. The full value is shown once, it carries the same permissions as an admin, and revocation is immediate and cannot be undone. See Authentication & Authorization.
append-only
A table whose rows are only inserted, never updated or deleted. usage.usage_events is append-only.
billing cycle
The recurring period an org is billed over; usage resets at the start of each cycle, and customers are never cut off mid-cycle. See Billing, Plans & Usage Metering.
break-glass account
When SSO is required, at least one owner kept able to sign in with a password in case the identity provider is unavailable. See Authentication & Authorization.
expand/contract migration
A two-deploy migration pattern (expand, then contract) used because pods roll one at a time, so old and new code briefly share the database — for example, add a nullable column and backfill (deploy 1), then drop the old column (deploy 2). See Database Migrations.
gateway
The single public entrypoint and the only authenticated edge; it runs verifySession then proxy-forwards to internal services, owns no schema, and listens on port 3000. See The Gateway & Request Flow.
included units
The usage quota bundled into a plan before overage is charged: Free 1,000, Team 25,000, Enterprise 500,000. See Billing, Plans & Usage Metering.
Incident Commander
The incident-response role that runs the incident, makes the calls, and keeps the timeline. See Incident Response & Postmortems.
internal service
A service not reachable from outside the cluster; it resolves at http://<name>.internal on container port 8080 and is never called directly by other services — cross-service reads go back through the gateway. See Platform Services.
invoice
A bill issued against an org, with the amount stored in cents. The status enum is draft | open | paid | void: draft while the cycle is in progress, open once finalized and sent, paid once charged, or void if cancelled. ("Overdue" describes a long-unpaid invoice but is not a status value.) See Billing, Plans & Usage Metering.
JIT provisioning (just-in-time)
When SSO is on, a first-time SSO user is added to the org automatically as a member. See Authentication & Authorization.
member
The lowest-privilege role (owner | admin | member) and the default role for JIT-provisioned SSO users. See Authentication & Authorization.
notification
An outbound message logged in notifications.notifications; the channel enum is email | webhook, the templates are invoice_receipt, member_invite, usage_alert, and payment_failed, and the status enum is pending | sent | failed. See Notifications & Webhooks.
org / organization
The top-level tenant; members, data, and billing all belong to an org (accounts.orgs). See Core Domain & Data Model.
overage
Usage above a plan's included units, billed on the next invoice as max(0, units - includedUnits) times the per-unit price. See Billing, Plans & Usage Metering.
owner
The highest role; exactly one per org, enforced by the unique partial index one_owner_per_org. See Authentication & Authorization.
plan
A billing tier; the name enum is free | team | enterprise. Free is $0 / 1,000 units; Team is $49 (4900 cents) / 25,000 units; Enterprise is $499 (49900 cents) / 500,000 units. See Billing, Plans & Usage Metering.
postmortem (blameless)
A blameless write-up after every Sev 1/2 incident, completed within two business days with action items tracked to done. See Incident Response & Postmortems.
readiness probe
The gateway's GET /healthz; it must not require auth and it gates traffic, so a bad image fails the rollout instead of taking the site down. See Release & Deployment.
role
A user's permission level; the enum is owner | admin | member, enforced server-side — the frontend hiding a control is convenience, not security. See Authentication & Authorization.
schema-per-service
Each service owns its own schema and tables inside one shared Postgres instance; no service reads another service's tables (wiki article 1001, ADR-002). See Architecture at a Glance.
session cookie
An opaque cookie named demo_session; the gateway looks it up (backed by Redis in production), checks expiry, and sets identity headers. A missing or expired cookie yields a 401. See The Gateway & Request Flow.
severity (Sev 1/2/3)
How bad an incident is: Sev 1 is a full outage or data-loss risk (page immediately, all hands); Sev 2 is a major feature down with no workaround (page on-call, open an incident); Sev 3 is degraded or partial with a workaround (business hours). See Incident Response & Postmortems.
SSO / SAML
Enterprise-plan-only sign-in via a SAML 2.0 identity provider (Okta, Entra ID, Google Workspace), set up at Settings → Security → Single sign-on. See Authentication & Authorization.
staging vs production
staging (cluster eks-staging, database rds-staging) deploys automatically on every merge to main; production (cluster eks-prod, database rds-prod) deploys from a tagged v* release after roughly a 30-minute soak. See Release & Deployment and Infrastructure & Environments.
trunk-based development
Branching off main, keeping PRs small with one review, and merging back; a merge to main deploys to staging automatically. See Code Review & Pull Requests.
unit
One metered action; usage above the plan's included units is billed as overage. See Billing, Plans & Usage Metering.
usage event
An append-only row in usage.usage_events (id, org_id, units, occurred_at), summed per org per period for billing. See Billing, Plans & Usage Metering.
webhook
A notification channel delivering events as JSON to a customer endpoint; add one at Settings → Notifications → Webhooks, verify each delivery via the Demo-Signature header, and note it is retried with backoff for up to 24 hours. The endpoint must return 2xx quickly or the delivery is treated as failed. See Notifications & Webhooks.
x-org-id / x-user-id
Identity headers the gateway sets from the verified session and forwards to internal services; services trust them and must not accept an orgId from the request body. See The Gateway & Request Flow and Security & Secrets.
