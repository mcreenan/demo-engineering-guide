The domain centers on the organization (org) — the top-level tenant that owns members, plans, invoices, usage, and notifications. Each service owns its own Postgres schema (Architecture at a Glance, ADR-002), so the tables below are grouped by the service that owns them, and this section is the canonical home for column-level schema detail.

In shortUUID primary keys (gen_random_uuid()); money as integer cents; timestamps as timestamptz in UTC. Every table except plans carries org_id. Only accounts.users → accounts.orgs is an enforced foreign key; other org_id links are logical and cross-schema.

Conventions

platform Architecture Decision Records (wiki article 1001)

These rules hold across every schema. They are not optional house style — the migrations enforce them and the TypeScript interfaces mirror them.

Schema-per-service
Every table lives in its owning service's schema inside one shared Postgres. No service reads another service's tables — see Platform Services for the ownership map.
UUID identifiers
Primary keys are uuid with default gen_random_uuid().
Money as integer cents
Amounts are stored as integer cents (e.g. amount_cents, monthly_price_cents) — never floats.
Timestamps
Time columns are timestamptz in UTC, commonly DEFAULT now().
Tenant scoping
Every table except plans carries org_id. Every service method is scoped to the org the gateway passes; a service must never trust an orgId from a request body. A missing org scope is a security bug — see Security & Secrets.

Entities at a glance

Six tables across four schemas. Only the User → Org edge is a declared Postgres foreign key (ON DELETE CASCADE); the other org_id relationships are logical, because services never cross schemas and resolve cross-service reads through the gateway.

accounts schema

platform/services/accounts/migrations/001_init.sql

The accounts schema holds the tenant root and its members. It requires CREATE EXTENSION citext so emails compare case-insensitively. The mirroring TypeScript symbols are Org, User, and Role.

TableColumnType / constraint
accounts.orgsiduuid PK, default gen_random_uuid()
nametext NOT NULL
created_attimestamptz NOT NULL DEFAULT now()
accounts.usersiduuid PK
org_iduuid NOT NULL REFERENCES accounts.orgs(id) ON DELETE CASCADE
emailcitext NOT NULL
roletext NOT NULL CHECK (role IN ('owner','admin','member'))
  • Email is unique within an org. UNIQUE (org_id, email) — case-insensitive via citext.
  • One owner per org. Enforced by a unique partial index: CREATE UNIQUE INDEX one_owner_per_org ON accounts.users (org_id) WHERE role = 'owner'.
  • First user owns the org. The first user of an org is its owner; transferring ownership reassigns the role and demotes the previous owner to admin. Role meaning and server-side enforcement live in Authentication & Authorization.

billing schema

platform/services/billing/migrations/001_init.sql

The billing schema holds the shared plan catalog and per-org invoices. TypeScript symbols: Plan and Invoice. Pricing, overage math, and invoice finalization are owned by Billing, Plans & Usage Metering — kept minimal here.

TableColumnType / constraint
billing.plansiduuid PK
nametext NOT NULL CHECK (name IN ('free','team','enterprise'))
monthly_price_centsinteger NOT NULL
included_unitsinteger NOT NULL DEFAULT 0
billing.invoicesiduuid PK
org_iduuid NOT NULL (plain uuid, no declared FK — billing is a separate schema/service)
amount_centsinteger NOT NULL DEFAULT 0
statustext NOT NULL DEFAULT 'draft' CHECK (status IN ('draft','open','paid','void'))
period_starttimestamptz NOT NULL
period_endtimestamptz NOT NULL
  • Shared catalog. plans has no org_id; it is the catalog of plan definitions. The seeded rows are free, team, and enterprise — pricing and included units are detailed in Billing, Plans & Usage Metering.
  • Invoice lookups. CREATE INDEX invoices_by_org ON billing.invoices (org_id, period_start DESC).
  • No FK on org_id. Invoices reference an org by id only; there is no database foreign key, because billing and accounts are owned by separate services that do not cross schemas.

Invoices move through a small state machine — from draft, to finalized, to charged, or cancelled:

draft
Billing cycle in progress.
open
Finalized and sent.
paid
Successful charge.
void
Cancelled.
Invoice status

  [*] --> draft
  draft --> open

"overdue" is not a statusThe Help Center may call a long-unpaid invoice "overdue", but overdue is not a value in the status enum. The only statuses are draft, open, paid, and void.

usage schema

platform/services/usage/migrations/001_init.sql

The usage schema records metered events. It is append-only — events are never updated or deleted. The TypeScript symbol is UsageEvent.

TableColumnType / constraint
usage.usage_eventsiduuid PK
org_iduuid NOT NULL
unitsinteger NOT NULL CHECK (units >= 0)
occurred_attimestamptz NOT NULL DEFAULT now()
  • Append-only. Rows are only ever inserted, giving an immutable event log.
  • Aggregate index. CREATE INDEX usage_events_by_org_time ON usage.usage_events (org_id, occurred_at) serves the billing access pattern of summing units for an org over a window.
  • Read through the gateway. Billing reads period totals from the usage service via the gateway — it never queries the usage schema directly. See The Gateway & Request Flow.

notifications schema

platform/services/notifications/migrations/001_init.sql

The notifications schema queues outbound messages for a delivery worker. TypeScript symbols: Notification, Channel, and Template. Channels and templates are owned by Notifications & Webhooks.

TableColumnType / constraint
notifications.notificationsiduuid PK
org_iduuid NOT NULL
channeltext NOT NULL CHECK (channel IN ('email','webhook'))
templatetext NOT NULL (no DB CHECK; valid values enforced in TypeScript)
statustext NOT NULL DEFAULT 'pending' CHECK (status IN ('pending','sent','failed'))
created_attimestamptz NOT NULL DEFAULT now()
  • App-enforced templates. Allowed values come from the TypeScript Template union — invoice_receipt, member_invite, usage_alert, payment_failed — not a database constraint.
  • Delivery index. A partial index serves the worker: CREATE INDEX notifications_pending ON notifications.notifications (created_at) WHERE status = 'pending'.
Changing the schemaAll schema changes go through migrations under platform/services/<name>/migrations, applied in filename order. Follow the expand-then-contract rule in Database Migrations; a service never migrates another service's schema.
