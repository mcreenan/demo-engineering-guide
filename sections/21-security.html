Security at Demo Co rests on two rules: the gateway is the only authenticated edge, and every query is scoped to the caller's org. Identity flows inward as x-user-id / x-org-id headers the gateway sets, and secrets live in AWS Secrets Manager — never in git.

  • One authenticated edge. The gateway verifies the session and sets identity headers; internal services trust those headers and never accept an orgId from a request body.
  • Org scoping is mandatory. Every query is scoped to the caller's org. A missing org scope is a security bug, not a nice-to-have.
  • Roles server-side. owner | admin | member are enforced on the server. The frontend hiding a button is convenience, not security.
  • Secrets in Secrets Manager. They reach pods as Kubernetes Secret env vars via secretKeyRef. Never commit a secret; rotate on a schedule and immediately on suspected exposure.
  • Report fast. Revoke first if you can, then report in #security. Sev 1 security issues follow the incident process.

The authorization model

Security & secrets (wiki article 1009) platform

The gateway is the only authenticated edge (ADR-001). It verifies the session, then sets x-user-id and x-org-id from the verified identity before forwarding inward. Internal services are not reachable from outside the cluster, so they trust those headers — but only those headers.

The single most important rule for backend code: the caller's org comes from the header the gateway set, never from the request. Internal services must never accept an orgId from a request body; they scope every query to x-org-id and ignore any body orgId. See proxy in .

Identity is set, not asked
The gateway runs verifySession, then copies the verified identity onto x-user-id / x-org-id. Downstream services do not re-read the cookie.
Every query is org-scoped
A missing org scope leaks another tenant's data. "A missing org scope is a security bug, not a nice-to-have."
Roles are enforced server-side
owner | admin | member checks live in the services. "The frontend hiding a button is convenience, not security."
Health check is the one exception
GET /healthz must not require auth; everything past it requires a valid session.
Never trust a body orgIdIf a handler reads the org from the request body instead of x-org-id, a caller can read or write another org's data. Trust the header the gateway set; ignore the body. Session lookup, expiry, and the 401 responses are detailed in Authentication & Authorization.

Secrets

Security & secrets (wiki article 1009) infra

Secrets live in AWS Secrets Manager and reach pods as Kubernetes Secret env vars, injected via secretKeyRef — see the manifests under infra/kubernetes/*.yaml in . The rules are simple: never commit a secret to git, rotate on a schedule, and rotate immediately on any suspected exposure.

The RDS master password is managed by Secrets Manager rather than hand-set: infra/terraform/rds.tf sets manage_master_user_password = true. No application code ever sees a literal database password.

A rotated secret looks like a crash loopIf a secret is missing or was just rotated, the affected service crash-loops on startup. Diagnose it with kubectl describe pod; this failure mode is also covered by Observability & On-Call.
Rotation cadence is not pinned downThe policy is "rotate on a schedule," but the sources give no concrete interval and describe no automated rotation tooling. Treat the specific schedule as undefined here and follow the current operational practice.

Product-facing keys & SSO

API keys SSO & SAML

Customers manage their own credentials, and the mechanics are owned by Authentication & Authorization. From a security standpoint the points that matter are the blast radius and the speed of revocation.

API keys
Created at Settings -> API keys (owners and admins only), shown in full once, and sent as a bearer token to https://api.demo.co (example prefix demo_sk_). A key has the same permissions as an admin, so store it in a secret manager — never source control.
Revocation
Revoke immediately if a key may have leaked. "Revocation takes effect at once and cannot be undone." Rotate by creating a new key, updating integrations, then deleting the old one.
SSO / SAML
Enterprise plan only; SAML 2.0 with Okta, Entra ID, or Google Workspace. Just-in-time provisioning adds a first-time SSO user as a member. When requiring SSO, keep one owner who can still sign in by password as a break-glass account.

Reporting a security issue

Security & secrets (wiki article 1009) Incident management (wiki article 1006)

If something looks wrong, act before you write it up. "Revoke first if you can, then report in #security." Every report is triaged, and Sev 1 security issues follow the incident process.

  1. Contain. Revoke the exposed credential or key first if you are able — revocation is immediate.
  2. Report. Post in #security with what happened and what you have already done.
  3. Escalate if severe. A Sev 1 security issue (full outage or data-loss risk) runs through Incident Response & Postmortems — declare, mitigate, communicate, resolve, postmortem.
When in doubt, ask"When in doubt, ask in #security." Over-reporting a non-issue is cheap; sitting on a real one is not.
