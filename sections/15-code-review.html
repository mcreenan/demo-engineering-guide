Code review keeps main releasable and spreads knowledge across the team. Treat it as a conversation, not a gate to grind through: branch off main, keep the pull request small, get one review, and merge.

In shortSmall PRs, the why in the description, tests with the change, a reply within a business day, and four things true before merge: tests pass and are covered, one approval, no unresolved “must fix,” and schema changes follow the migrations runbook.

PRs span all three repositories — , , and — and the same expectations apply everywhere.

Why we review

Code review (wiki article 1005)

Two goals, in order: keep main releasable, and spread knowledge so no part of the system has only one person who understands it. The point is a shared, deployable codebase — not a checkpoint to survive.

For authors

Code review (wiki article 1005)

Make the change easy to review and easy to trust.

  • Keep it small. One idea per PR; split a big change into reviewable pieces.
  • Write the why. Put the reason for the change in the description, along with how you verified it.
  • Self-review first. Read your own diff before asking anyone else to.
  • Bring the tests. Tests come with the change, not in a follow-up — see Testing.

For reviewers

Code review (wiki article 1005)

Respond within a business day — a stuck PR blocks a teammate. Review for substance and let tooling handle the rest.

What to review for
Correctness, clarity, tests, and fit with the surrounding code.
What not to review for
Style — let the linter handle it (eslint via make lint; see Coding Standards & Conventions).
How to comment
Be specific and kind, and mark each comment “must fix” or “nit” so the author knows what blocks the merge.

The bar to merge

Code review (wiki article 1005) .github/workflows/deploy.yml

All four of these must be true before a PR merges:

  1. Tests pass and are covered. CI runs the test suite on every push as the test job, and it gates everything downstream.
  2. One approval. A single review is enough to merge.
  3. No unresolved “must fix.” Nits can wait; must-fixes cannot.
  4. Schema changes follow the migrations runbook. Any schema change goes through the process in Database Migrations (wiki article 1007).

CI enforces the test gate in .github/workflows/deploy.yml: the build-and-push job declares needs: test, so a red build never deploys. The same suite runs locally per repo:

Why the bar is highBecause merges deploy to staging immediately, the bar is “good enough to be live for customers tomorrow.” A merge to main runs test, builds images, and deploys to staging automatically — see Release & Deployment.

The review loop

A PR moves from open to merged with one back-edge: reviewer comments send it back to the author until the bar is met. The “one review” of trunk-based development is the same as the “one approval” on the merge bar.
