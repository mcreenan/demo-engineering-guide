Every request enters through one door: the gateway. It is the only publicly-exposed service and the only authenticated edge — it verifies the session, attaches the caller's identity as headers, and forwards the request to an internal service that does the real work and returns JSON.

  • One entrypoint. The browser calls https://api.demo.co/<service>/...; internal services are never reachable directly.
  • Auth at the edge. The gateway runs verifySession and rejects anything without a valid, unexpired session with a 401.
  • Identity by header. The gateway forwards to http://<name>.internal, adding x-user-id and x-org-id from the verified session.
  • Trust the headers. Internal services scope every query to x-org-id and must never accept an orgId from the request body.
  • Cross-service reads loop back. Services never call each other directly — a read goes back out through the gateway.

The request lifecycle, end to end

platform Architecture Decision Records (wiki article 1001)

A single request travels through four steps. The gateway owns steps 1–3; the internal service owns step 4 and hands its result back through the gateway.

  1. Browser sends the cookie. The frontend calls https://api.demo.co/<service>/... carrying the session cookie demo_session. The api-client sends it with credentials: "include" — see Frontend Development.
  2. Gateway verifies the session. verifySession reads the opaque demo_session cookie and resolves it to identity. A missing token returns 401 {"error":"no session"}; an expired or unresolved session returns 401 {"error":"session expired"}. On success it stashes the resolved session (userId, orgId, expiresAt) on the request and continues.
  3. Gateway proxies inward. The proxy factory forwards the request to http://<name>.internal, preserving the method and adding identity headers x-user-id and x-org-id.
  4. Service responds. The internal service does its work scoped to that org, returns JSON, and the gateway relays the upstream status and body straight back to the caller.

Step 2 — verifySession

platform

The gateway is the only place auth happens. verifySession turns an opaque cookie into a verified identity, or rejects the request outright. Session mechanics, API keys, and SSO are owned by Authentication & Authorization.

Reads
The opaque cookie demo_session from the request.
Missing token
Returns 401 {"error":"no session"}.
Unresolved or expired
When no session resolves, or expiresAt is in the past, returns 401 {"error":"session expired"}.
On success
Stashes the resolved session — a userId, orgId, and expiresAt — on the request and proceeds.
Lookup
lookupSession resolves a token to identity; it is backed by Redis in production. The sample implementation returns null.

Step 3 — proxy

Once the session is verified, proxy relays the request inward. Each public prefix is mounted with proxy(<name>), and the request is forwarded to that service's internal address.

The trust boundary

platform Security & secrets (wiki article 1009)

Because the gateway is the only authenticated edge, internal services trust the identity headers it sets. That trust comes with one hard rule, which Security & Secrets owns as policy.

Never accept orgId from the bodyDownstream services scope every query to the x-org-id header and must ignore any orgId in the request body. A missing org scope is a security bug.

The /healthz probe

platform infra

One route is intentionally unauthenticated. GET /healthz returns { ok: true } and is registered before the verifySession middleware, so it never requires a session — everything mounted after that line does.

In Kubernetes the gateway Deployment uses /healthz on port 3000 as its readiness probe (with initialDelaySeconds: 5), which gates traffic during a rollout. The ALB Ingress detail is owned by Infrastructure & Environments.

Why it must stay openThe probe is the liveness/readiness check for Kubernetes — if it required auth, the cluster could not tell whether a pod was healthy.

Cross-service reads loop back through the gateway

platform

Internal services never call each other directly and never read another service's schema. When one service needs data from another, it comes back out through the gateway over HTTP — the same authenticated path a browser uses.

For example, BillingService reads usage counters by calling GET /usage/total?period=current through the gateway, never the usage schema directly. The overage and metering mechanics live in Billing, Plans & Usage Metering.

Why one gateway

Architecture Decision Records (wiki article 1001)

The single-entrypoint design is recorded as ADR-001, "API gateway as the single entrypoint": all external traffic flows through one gateway, internal services are not publicly reachable, and there is one place for auth, rate limiting, and audit logging. The schema-per-service decision is ADR-002. The at-a-glance topology lives in Architecture at a Glance.

ADR numberingThe wiki (article 1001) is canonical for ADRs and numbers the gateway decision ADR-001. The platform doc architecture.md swaps the numbers; follow the wiki. Rate limiting and audit logging are listed as gateway responsibilities in the ADR, but are not implemented in the current gateway code — treat them as intent, not behavior.
