Demo Co runs on AWS in us-east-1: one Terraform configuration in describes two environments — staging and production — each an EKS cluster plus a shared Postgres instance, selected by Terraform workspace.

In shortOne Terraform config, two workspaces (staging | production); remote state in S3 with a DynamoDB lock; never terraform apply to production from a laptop — prod applies go through CI; EKS 1.29; one shared Postgres 16, schema-per-service; the gateway is the only thing on the public internet (an ALB Ingress at api.demo.co), everything else lives in private subnets.

This section owns cloud, Kubernetes, and Terraform — how the system is run. The topology at a glance

infra/terraform, infra/kubernetes

Everything lives inside one VPC (10.0.0.0/16). Only the gateway's load balancer is reachable from the internet; the cluster nodes and the database sit in private subnets, and the database accepts connections only from the cluster.

Internal services never accept outside traffic and resolve only inside the cluster as <name>.internal — a cross-service read goes back out through the gateway (the proxy in resolves those names). The gateway-as-single-entrypoint rationale is owned by Architecture at a Glance.

Terraform: one config, two workspaces

infra/terraform/main.tf, infra/README.md

Terraform requires version >= 1.6 with the AWS provider ~> 5.0. There is one config for both environments; the workspace picks which one you are operating on, and the code reads it as local.env = terraform.workspace — one of two values.

State
Remote, in S3: bucket demo-co-tfstate, key platform/terraform.tfstate, encrypt = true, with a DynamoDB lock table demo-co-tflock. One state file per environment, selected by workspace.
Default tags
Applied to every resource: Owner = platform-team, Environment = terraform.workspace, ManagedBy = terraform.
Naming
Resources are named from local.name = "demo-<env>"; a local.is_prod flag (env == "production") drives the prod-only sizing and safety toggles below.

The local workflow is read-only: select the workspace, plan, and review. The repo is laid out by concern:

terraform/
The config — main.tf, network.tf, eks.tf, rds.tf.
kubernetes/
The manifests applied to the cluster by the deploy pipeline.
.github/workflows/deploy.yml
The CI pipeline (owned by Release & Deployment).
runbooks/
Operational runbooks — deploys.md, incident-response.md.

RDS: one shared Postgres per environment

infra/terraform/rds.tf, infra/terraform/main.tf

Each environment has a single Postgres instance (rds-<env>, engine postgres version 16) shared by every service, with one schema per service. Column-level schema is owned by Core Domain & Data Model; this section covers the instance itself.

The prod-only behaviors all key off the local.is_prod flag — multi-AZ, deletion protection, and the final-snapshot decision are derived from it:

  • Multi-AZ. local.multi_az = is_prod — on in prod, off in staging.
  • Final snapshot. skip_final_snapshot = !is_prod — staging skips it, prod takes one on destroy.
  • Deletion protection. Enabled in prod only.

The instance database and master user are both named platform, and manage_master_user_password = true — so the credential is created and stored in AWS Secrets Manager rather than set in the config. The database sits in a subnet group spread across the private subnets.

Networking: public edge, private everything

infra/terraform/network.tf

The VPC is a 10.0.0.0/16 network with DNS hostnames enabled. It has two public subnets and two private subnets, each pair spread across two availability zones (chosen dynamically), so the only thing exposed to the internet is the load balancer.

Public subnets — for the load balancer
10.0.0.0/24 and 10.0.1.0/24, with map_public_ip_on_launch set. These host the ALB that fronts the gateway.
Private subnets — for the cluster and database
10.0.10.0/24 and 10.0.11.0/24. The EKS node group and the RDS instance live here; neither is internet-reachable.
Database security group
Ingress allows Postgres (5432/tcp) only from the cluster security group — the database is reachable only from the cluster, nothing else.
What the sources show — and don'tnetwork.tf defines the VPC, subnets, and the db and cluster security groups. It does not include NAT gateway, internet gateway, or route-table resources, so the exact egress mechanics for the private subnets are not described here.

Kubernetes manifests

infra/kubernetes/gateway.yaml, accounts.yaml, billing.yaml

The manifests under kubernetes/ are applied to the cluster by the deploy pipeline. There are two shapes: the public gateway (a Deployment plus a Service plus an ALB Ingress) and internal services (a Deployment plus an internal Service, no Ingress).

Gateway — the only public exposure

The gateway is a Deployment of three replicas that is the only workload with an Ingress:

Deployment
replicas: 3, labels app: gateway, tier: edge, listening on container port 3000.
Readiness probe
httpGet /healthz on port 3000, initialDelaySeconds: 5.
Service
Maps port 80 to target port 3000.
Ingress
Annotated kubernetes.io/ingress.class: alb and alb.ingress.kubernetes.io/scheme: internet-facing; host api.demo.co, path / (Prefix) routes to gateway:80. It terminates TLS for that host.
Session store
SESSION_REDIS_URL comes from a secretKeyRef — Secret gateway-secrets, key redis-url.

Internal services — the repeated pattern

Internal services all follow one shape and resolve only as <name>.internal inside the cluster:

  • Deployment. replicas: 2, label tier: internal.
  • Port. Listens on container port 8080; the Service maps 80 to 8080.
  • No Ingress. Never exposed to the internet — only the gateway is.

The two internal manifests present in the sources are accounts.yaml and billing.yaml:

The usage and notifications services are expected to follow the same internal pattern (2 replicas, internal Service, no Ingress, <name>.internal), but their manifests are not in the provided files — don't rely on per-service detail for them. Likewise, only the gateway declares a readiness probe; internal-service probes are not shown.

Secrets reach pods as env vars

Every secret is injected as an environment variable through a Kubernetes secretKeyRef — the manifest names a Secret and a key, and Kubernetes binds the value into the container env:

That is the delivery mechanism only. The policy — AWS Secrets Manager as the source of truth, rotation on a schedule, and never committing a secret to git — is owned by Security & Secrets.

Manifests pin :latest; the pipeline ships SHA tagsThe committed manifests reference :latest images, while the deploy pipeline builds and pushes images tagged with the commit SHA. How kubectl apply reconciles the SHA tag into the manifest at deploy time is not shown in the sources — for the deploy mechanics see Release & Deployment.
