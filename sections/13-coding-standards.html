A handful of conventions keep two repos and five services consistent. Store money as integer cents and ids as UUIDs; scope every backend query to the caller's org and never trust an orgId from the request body; reach the backend only through api-client and build UI only from ui; keep pull requests small and let the linter own style.

  • Data. UUID ids, money in integer cents, timestamps as timestamptz in UTC, inputs validated with zod.
  • Backend. Org-scope every query, ignore any body orgId, never read another service's schema.
  • Frontend. All backend calls go through api-client; all primitives come from ui.
  • Process. One idea per pull request; review for correctness and tests; let eslint handle style.

This section is a checklist with cross-links. The deep detail lives in the owning sections — schema in Core Domain & Data Model, services in Backend Development, the UI rules in Frontend Development, and merge process in Code Review & Pull Requests.

Backend conventions

platform Security & secrets (wiki article 1009)

Services are classes; their methods are async and typed. Three rules matter most, and the first one is a security rule, not just a style one — a missing org scope is a bug. See Backend Development and Security & Secrets.

Org-scope every query
Every method takes an orgId and scopes its SQL to it — e.g. listMembers runs SELECT * FROM users WHERE org_id = $1 and invoicesForOrg filters on WHERE org_id = $1.
Never trust a body orgId
The verified org comes from the session via the gateway, which injects x-org-id and x-user-id. Downstream services must ignore any orgId in the request body. See The Gateway & Request Flow.
No cross-service DB reads
A service never queries another service's schema. Cross-service data goes back out through the gateway over HTTP — billing reads usage totals from the usage service that way, never from the usage schema directly.
A missing org scope is a security bugTreat an unscoped query the same as a vulnerability, not a code-style nit. The gateway is the only thing that establishes identity; once a request is inside a service, the only trustworthy org is the one in x-org-id.

Frontend conventions

web

The frontend has no database access and no knowledge of internal services — the gateway is the only backend. Two rules keep apps thin; both are owned in depth by Frontend Development.

All backend access through api-client
Apps never call fetch directly. packages/api-client is the only place that knows endpoint URLs and shapes. If an endpoint isn't in the client, add it there first.
All primitives from ui
Apps never hand-roll buttons, tables, inputs, badges, or design tokens — they come from packages/ui's public index. An app that needs a new primitive contributes it upstream rather than forking one locally.

The shared pieces have names worth knowing: api-client exports the typed api wrapper and throws SessionExpiredError on a 401, while ui exports Button, Table, and the badges.

Auth follows one pattern: the session cookie is sent with credentials: "include", and a 401 surfaces as SessionExpiredError. The gateway base URL resolves from NEXT_PUBLIC_API_URL (default http://localhost:3000). Session details are owned by Authentication & Authorization.

Process and style ownership

Code review (wiki article 1005) platform web

Two process rules keep changes reviewable. Keep pull requests small — one idea per PR, and split big changes. Review for correctness, clarity, tests, and fit, and let the linter handle style instead of spending review comments on it. The full review and merge bar is owned by Code Review & Pull Requests, and tests-come-with-the-change is covered in Testing.

Style is enforced by tooling, not opinion. Linting is eslint, run as a script in both repos; typing is TypeScript, with a build-mode typecheck on the backend.

The only style tooling is eslintThere is no separate formatter in either repo — eslint and the TypeScript compiler are the whole story. Defaults like UUID ids, timestamptz timestamps, and _cents columns are also enforced where they belong, in migrations; see Database Migrations.
