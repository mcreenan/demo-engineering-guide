We test with vitest across every workspace, in both the backend and frontend repos. Run the suite locally before you push; CI runs it on every push to main, and a red build never deploys.

In shortOne runner everywhere (vitest, ^1.6.0). Locally: make test in platform, pnpm test in web. In CI: the deploy pipeline’s test job gates build and deploy. Tests come with the change, and “tests pass and are covered” is the first item on the merge bar.

The same runner spans all three repos — , , and (which owns the CI workflow). Both code repos require Node 20 or newer.

The test runner

platform/package.json web/package.json

There is exactly one test runner to learn: vitest, pinned to ^1.6.0 as a devDependency in both repos. In every case the underlying command is vitest run, so npx vitest run works directly in any workspace if you want to skip the wrapper script.

Running tests locally

platform/Makefile web/package.json

Each repo wraps the runner in the command that fits its toolchain. Both run the suite across all of that repo’s workspaces.

RepoCommandWhat it does
make testRuns npx vitest run across all service workspaces (also the npm script test).
pnpm testRuns vitest run across the web workspaces.
eithernpx vitest runThe underlying command both wrappers call.
 
# platform
make test

# web
pnpm test

# either, directly
npx vitest run

These commands are introduced alongside the rest of the local workflow in Local Development. For where vitest fits in the backend toolchain (next to tsc --build and eslint), see Backend Development; the frontend equivalent lives in Frontend Development.

Tests in CI

infra/.github/workflows/deploy.yml Release & deployment (wiki article 1003)

The deploy pipeline in begins with a test job that installs dependencies and runs the suite across the platform workspaces, on Node 20.

Command
npm ci then npm test.
Triggers
Every push to main and every v* tag.
Gating
The build-and-push job declares needs: test, and deploy declares needs: build-and-push — so a failing test job stops the pipeline before any image is built or shipped. A red build never deploys.

Scope of the CI gateThe pipeline described here runs the platform test suite. The full build-and-deploy story — staging soak, tagged production releases, and rollback — is owned by Release & Deployment.

Expectations

Code review (wiki article 1005)

Testing is an author responsibility, not a follow-up task.

  • Tests come with the change. Write them as part of the PR and self-review your own diff first.
  • “Tests pass and are covered” is the first item on the bar to merge — see Code Review & Pull Requests.
  • Reviewers check tests. They review for correctness, clarity, tests, and fit.
