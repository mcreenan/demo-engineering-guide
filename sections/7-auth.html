Authentication answers who you are; authorization answers what you may do. At Demo Co both are settled at the gateway and in the services behind it: an opaque demo_session cookie identifies the caller, and a server-side role decides what that caller can do inside their org.

  • Sessions. The browser sends an opaque demo_session cookie; the gateway verifies it, checks expiry, and attaches identity headers before forwarding inward.
  • Roles. owner | admin | member, enforced server-side. The frontend hiding a control is convenience, not security.
  • One owner per org. Enforced by a unique partial index; transferring ownership demotes the previous owner to admin.
  • API keys. Bearer tokens (demo_sk_) with admin-level permissions, shown once, revocable instantly.
  • SSO/SAML. Enterprise plan only; first-time SSO sign-in provisions a member; keep a break-glass owner with a password.
Where this livesThis section owns auth, sessions, API keys, and SSO. The request lifecycle and header injection are owned by The Gateway & Request Flow; security policy by Security & Secrets; plan gating for SSO by Billing, Plans & Usage Metering.

Sessions: the demo_session cookie

platform infra Security & secrets (wiki article 1009)

Authentication is a single, opaque cookie. The browser sends demo_session; the gateway reads it via req.cookies["demo_session"] and resolves it to a verified identity. The cookie value is opaque — it carries no claims itself, so the gateway must look it up.

verifySession is the gateway's one job before routing: look up the token, confirm it has not expired, and stash the identity on the request so proxy can forward it. The resolved Session is { userId, orgId, expiresAt }.

No cookie
Returns 401 {"error":"no session"}.
Missing session or expired
When the token resolves to nothing, or expiresAt < now, returns 401 {"error":"session expired"}.
Valid
Stashes the Session on the request and continues to the proxy.
Lookup
lookupSession resolves a token to identity; its commented query is SELECT user_id, org_id, expires_at FROM sessions WHERE token = $1. It is backed by Redis in production and is not implemented in the sample.

The Redis backing is confirmed by infra: the gateway pod receives a gateway-secrets/redis-url secret via secretKeyRef — see . After verifying, the gateway forwards to the internal service and sets the identity headers from the verified session, as detailed in The Gateway & Request Flow.

Session lifetime is not specifiedThe gateway only checks expiresAt < now; no default session TTL or lifetime is defined in the sources. How sessions are created (the login endpoint, credential verification) is also not present here — only verification is shown.
Trust the headers, not the bodyInternal services trust x-user-id and x-org-id, never re-read the cookie, and must ignore any orgId in the request body. This policy is owned by Security & Secrets.

Roles & authorization

platform web Members & roles Security & secrets (wiki article 1009)

There are exactly three roles, and authorization is always enforced server-side. The dashboard may hide a control the caller cannot use, but that is convenience, not security — the check that matters runs in the service.

RoleCan doCannot do
ownerEverything, including billing and transferring ownership.—
adminInvite/remove members, change roles, manage settings.—
memberUse the product.Manage billing or other members.

In the frontend, MembersPage derives canInvite from the role (owner or admin) and renders a RoleBadge; the authoritative check still happens in AccountsService. The column-level schema for accounts.users — the role CHECK and the one-owner index — is owned by Core Domain & Data Model.

Exactly one owner per org

Every org has exactly one owner, and the database guarantees it. A unique partial index one_owner_per_org ON accounts.users (org_id) WHERE role = 'owner' makes a second owner impossible. The first user of an org is always its owner — createOrg inserts that user with role owner.

  • Inviting. Only owners and admins may invite. inviteMember enforces it server-side; the dashboard mirrors it with canInvite.
  • Changing roles. setRole updates a member's role within the org.
  • Removing a member. Access ends immediately; anything they created stays with the org.
Transferring ownership before someone leavesTo hand off an org, promote another member to owner. The previous owner is demoted to admin so the org always has exactly one owner — do this before the current owner leaves, not after.

API keys

API keys Security & secrets (wiki article 1009)

API keys let an integration call the API without a browser session. They are created at Settings → API keys by owners and admins only, and a key has the same permissions as an admin in your org — so treat it like a password.

Created at
Settings → API keys (owners and admins only).
Shown once
The full value appears only at creation — "Copy the key now." Store it in a secret manager, never in source control.
Used as
A bearer token sent to https://api.demo.co, e.g. Authorization: Bearer demo_sk_... (prefix demo_sk_).
Permissions
The same as an admin in your org.
Revoked at
Settings → API keys. Revocation takes effect at once and cannot be undone.

To rotate a key, create a new one, update your integrations to use it, then delete the old key. Because customer-facing keys can be revoked instantly, revocation is the first move on suspected exposure — secret-handling policy is owned by Security & Secrets.

# Call the API with a key instead of a session cookie
curl https://api.demo.co/accounts/members \
  -H "Authorization: Bearer demo_sk_..."

SSO & SAML

Single sign-on (SAML)

Single sign-on is available on the Enterprise plan only and supports SAML 2.0 identity providers — Okta, Entra ID, Google Workspace, and other SAML 2.0 providers. Plan details and pricing are owned by Billing, Plans & Usage Metering.

  1. Open the settings. An owner or admin goes to Settings → Security → Single sign-on.
  2. Configure the IdP. Copy the ACS URL and Entity ID into the IdP's SAML app.
  3. Return the metadata. Paste the IdP metadata URL back (or upload the XML) and Save.
  4. Test before enabling. Use Test connection before turning SSO on.

With just-in-time provisioning, a first-time SSO sign-in adds the user to the org automatically as a member; promote them later if needed. Once SSO is verified you can require it, after which members can no longer sign in with a password.

Keep a break-glass ownerWhen you require SSO, keep at least one owner who can still sign in with a password as a break-glass account in case the IdP is unavailable. This is help-documented product behavior; the sources show no code enforcement of it.

When a session expires

web Troubleshooting login

When the gateway returns a 401, the frontend surfaces it as a re-login. The api-client sends the cookie with credentials: "include" and, on res.status === 401, throws a SessionExpiredError — the api-client is the only backend wrapper, as covered in Frontend Development.

For users, the Help Center frames repeated logouts as a timed-out session or a browser blocking cookies: sign in again and enable cookies for demo.co. Private or incognito windows each start a fresh session, and a password reset link is valid for one hour.

  • SSO users. Use "Sign in with SSO" — a password may not exist for the account.
  • SSO failing. Confirm IT has assigned the Demo Co app, and ask an admin to re-run Test connection.
