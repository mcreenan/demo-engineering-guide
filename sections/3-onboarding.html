The system in one paragraph

Demo Co runs on three repositories. is the backend: Node + TypeScript services behind a single gateway, on one Postgres with a schema per service. is the frontend: Next.js apps built on shared ui and api-client packages. holds the Terraform, Kubernetes manifests, and the deploy pipeline.

The browser only ever talks to the gateway. The gateway authenticates every request and routes it to an internal service; internal services are not reachable from outside. For the full picture and the diagram, see Architecture at a Glance.

Set up your machine

Engineering onboarding (wiki article 1004) platform web

Four steps get you to a running stack. The deep details — DATABASE_URL, the Node version, and the seed fixture — live in Local Development.

  1. Clone all three repos. , , and .
  2. Boot the backend. In platform, run make dev. It migrates and seeds a local Postgres, then starts every service with hot reload.
  3. Boot the frontend. In web, run pnpm install && pnpm dev. The dashboard comes up on :3000 (and the admin console on :3001).
  4. Join the channels. #engineering and #incidents.
# platform — migrate + seed, then start every service
make dev

# web — all apps in parallel (dashboard :3000, admin :3001)
pnpm install && pnpm dev

# web — just one app
pnpm --filter dashboard dev
Requirementsmake dev needs DATABASE_URL set and Node 20 or newer. The seed loads one demo org, a few users, and the three plans, so the app has data to render on first run.

Ship your first change

Engineering onboarding (wiki article 1004) Code review (wiki article 1005) Release & deployment (wiki article 1003)

Demo Co uses trunk-based development: branch off main, keep the PR small, get one review, and merge. A merge to main triggers CI, builds container images, and deploys to staging automatically — so every merge needs to be good enough to be live for customers tomorrow.

The bar to merge

A PR is ready to merge when it clears these. Each one is owned by a deeper section — follow the links for the full story.

RequirementWhere it lives
Tests pass and are covered (they come with the change)Testing
One approval; no unresolved “must fix” commentsCode Review & Pull Requests
Schema changes follow the migration runbookDatabase Migrations

When authoring a PR: keep it to one idea, write the why and how you verified it, and self-review before asking for eyes. The full norms are in Code Review & Pull Requests.

After staging, production is promoted manually following a soak of roughly 30 minutes. The pipeline, promotion, and rollback all belong to Release & Deployment.
