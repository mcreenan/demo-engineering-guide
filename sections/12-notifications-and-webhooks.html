Demo Co notifies people by email and notifies systems by webhook. Both go through one service, the NotificationsService in : a service enqueues a notification through the gateway, then a background worker delivers it and records the outcome, retrying failures with backoff.

  • Two channels. email for people, webhook for systems.
  • Four templates. invoice_receipt, member_invite, usage_alert, payment_failed.
  • Always-sent vs. optional. Receipts, invites, and payment-failed emails can't be turned off; usage alerts are a toggle.
  • Async by design. enqueue returns immediately; deliverPending does the actual sending.
  • Webhooks. Register an endpoint, choose events, verify the Demo-Signature header, return 2xx quickly.

The delivery model

platform Notifications & webhooks

Delivery is split into two halves so that enqueuing is fast and sending is resilient. Producing services never deliver inline; they hand off a row and move on.

enqueue(orgId, template, channel = "email")
Inserts a row with status pending and returns immediately; the channel defaults to email.
deliverPending
The worker tick. It claims pending rows, sends them, and marks each sent or failed; failures are retried with backoff.

A row's status moves through pending → sent or pending → failed. The worker claims work safely with row-level locking so two workers never grab the same row:

-- deliverPending claims a batch of due work
SELECT * FROM notifications.notifications
WHERE status = 'pending'
FOR UPDATE SKIP LOCKED;

A partial index, notifications_pending, keeps that poll cheap by indexing only the pending rows. The full column-level schema is owned by Core Domain & Data Model.

Webhooks

Notifications & webhooks

Webhooks let a customer's systems react to events. Set them up at Settings → Notifications → Webhooks. Events are delivered as JSON.

  1. Add an endpoint URL. Register where deliveries should be sent.
  2. Choose which events to send. Pick from invoices, usage alerts, and member changes.
  3. Verify each delivery. Check the signature in the Demo-Signature header before trusting the payload.

A Recent deliveries view on the same settings page shows what was sent and the response received, which is the first place to look when an endpoint isn't reacting.

Return 2xx quickly or it's a retryYour endpoint must return a 2xx response fast. Slow or error responses are treated as failures and retried with backoff for up to 24 hours. Do heavy work asynchronously and acknowledge first.
Always verify the signatureTreat every payload as untrusted until you've verified the Demo-Signature header. Signature verification and secret handling are owned by Security & Secrets.
Not documented hereThe sources don't specify the signature algorithm, the exact backoff schedule, the worker's polling interval, the webhook payload shape, or a canonical list of event identifiers (only the categories invoices, usage alerts, and member changes). Don't assume values that aren't in the Help Center or the code.
