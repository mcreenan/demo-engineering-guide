The frontend lives in the repo: a pnpm monorepo of two Next.js apps (a customer dashboard and an internal admin console) built on two shared packages (a component library and a typed gateway client). Every app reaches the backend through one client and reuses one set of UI primitives — apps own routes and page composition, the packages own everything reusable.

  • Two apps. apps/dashboard is the signed-in customer app (billing, members, usage) on dev port :3000; apps/admin is the internal Demo Co staff console on :3001.
  • Two packages. @demo/ui holds shared components and design tokens; @demo/api-client is the one typed wrapper over the gateway.
  • Three rules of the road. Apps never call fetch directly, never hand-roll UI primitives, and talk only to the gateway — no direct database access.
  • Source of truth. This repo is authoritative for what the product looks like and how the UI behaves; for business rules (what a plan includes, how an invoice is computed) the repo wins.
  • Stack. Next.js ^14.2, React ^18.3, pnpm 9, Node >= 20.

Rules of the road

web

Three conventions keep the apps thin and the shared layer authoritative. They are the heart of this section.

  1. Apps never call fetch directly. All backend access goes through @demo/api-client, the single typed map of the gateway's endpoints. If an endpoint isn't in the client, add it there first.
  2. Apps never hand-roll UI primitives. Buttons, tables, inputs, and design tokens live in @demo/ui. An app that needs a new primitive contributes it upstream rather than forking one locally.
  3. The gateway is the only backend. The frontend has no direct database access and no knowledge of internal services — it sees one API surface at https://api.demo.co. Auth is a session cookie set during login.
Apps own routes; packages own reuseAn app's job is routing and page-level composition. Anything reusable — an endpoint shape, a component, a token — belongs in a package so it has exactly one definition.

Data flow for a page

web

A typical page never sees a URL or a raw response. It calls a hook, the client talks to the gateway, and the page renders the typed result with shared components. The billing page is the representative example.

  1. Page calls a hook. A page in apps/dashboard renders and calls a hook from the client, e.g. useInvoices.
  2. Client issues the request. The api-client sends GET /billing/invoices against the gateway with the session cookie (credentials: "include").
  3. Gateway authenticates and routes. The gateway verifies the session and forwards to the billing service — request-lifecycle detail is owned by The Gateway & Request Flow.
  4. Page renders. The result comes back typed, and the page renders it with components from @demo/ui (Table, Money, StatusBadge).

The api-client: the one place that knows the gateway

web

@demo/api-client is the only place in the frontend that knows endpoint URLs and shapes. Its public surface exports the typed api object, the SessionExpiredError, the types, and a hook per endpoint — apps import these and never call fetch themselves.

Base URL
process.env.NEXT_PUBLIC_API_URL ?? "http://localhost:3000" — the gateway base, defaulting to the local dev gateway.
Auth
The session cookie, sent automatically because every request uses credentials: "include".
Fetch wrapper
get calls fetch(BASE + path, { credentials: "include" }). A 401 throws SessionExpiredError; any other non-ok response throws an Error of the form ${path} → ${status}.
Typed methods
api.listInvoices() → GET /billing/invoices; api.listMembers() → GET /accounts/members.
Hooks
useInvoices, useMembers, useInviteMember, useOrgSearch.

Every hook follows the same pattern that useInvoices demonstrates: call the typed client, then expose { data, isLoading, error }. The other hooks mirror that shape.

Hook implementations are elided in the sampleThe sample sources note that real hooks use the app's data-fetching library, which is elided here. The library is not named in the available sources, so treat the concrete implementation as an unknown — only the { data, isLoading, error } contract is given. Session expiry and the demo_session cookie are owned by Authentication & Authorization.

Typed shapes mirror the backend enums

The client declares the shapes it returns, and their enum values match the backend exactly. The badge helpers in the UI library render these same enums, so labels never drift from the data model.

TypeFields
Invoiceid, amountCents, status (draft | open | paid | void), periodStart, periodEnd
Memberid, email, role (owner | admin | member), isCurrentUser

The column-level schema and the enum definitions these mirror are owned by Core Domain & Data Model; invoice statuses and overage are owned by Billing, Plans & Usage Metering.

The ui package: one button, one table, shared badges

web

@demo/ui exports the shared primitives every app imports. Variants, sizing, and badge colors all come from the design tokens — do not reimplement these inside an app.

Button
The one button. Props: children, onClick?, href?, variant? (primary | secondary | ghost), disabled?. Renders an <a> when href is set, otherwise a <button>.
Table
The shared data table. Props: columns, rows, empty?. It renders the empty text in place of the body when there are no rows.
Money
Renders integer cents as $${(cents/100).toFixed(2)} — cents in, formatted out. Money never uses floats for storage.
StatusBadge
The invoice status pill for draft | open | paid | void. Colors come from tokens; labels match the backend.
RoleBadge
The member role pill for owner | admin | member.
Design tokensButton variants, sizing, and badge colors are driven by design tokens referenced as tokens.css. The actual token names and values are not present in the available sources — treat them as an unknown rather than inventing names.

Representative pages

web

Three pages show the conventions in practice — each pulls data through a hook and renders it with shared components.

Dashboard billing

apps/dashboard/src/pages/billing.tsx uses useInvoices plus Table, Money, and StatusBadge. It is read-only and mirrors the billing service's invoice model — the status values shown are exactly the backend's.

Columns
Period, Amount, Status.
Statuses shown
draft, open, paid, void — exactly the backend enum.
Empty state
“No invoices yet — your first one is generated at the end of the cycle.”
Out of scope
Plan changes and card updates live on the Settings → Billing route, not this page.

Dashboard members

apps/dashboard/src/pages/members.tsx uses useMembers and useInviteMember with Table, Button, and RoleBadge. Roles match the accounts service — owner, admin, member — and render through RoleBadge. The invite control is shown only when canInvite = role === "owner" || role === "admin".

Hiding a control is convenience, not securityThe UI hides the invite button from plain members, but the backend enforces the same rule — so the hidden button is convenience only. Roles are enforced server-side; see Authentication & Authorization.

Admin home

apps/admin/src/pages/index.tsx uses useOrgSearch with Table and Button so Demo Co staff can look up a customer org for support tickets. Columns: Org, Plan, Members, plus a trailing action column whose Button opens the org.

It is internal only: access is gated behind staff SSO at the gateway. The admin app reads the same API as the customer dashboard but with a staff session that can act across orgs, and it never touches the database directly. The staff-session and SSO mechanics are owned by Authentication & Authorization.
