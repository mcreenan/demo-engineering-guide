Demo Co ships from main continuously. Every merge to main runs CI, builds container images, and deploys to staging automatically; production is a tagged release — push a git tag matching v* and the same pipeline deploys to production.

This section owns the deploy pipeline and rollback. The short version is below; the rest of the section is the detail.

  • Merge to staging is automatic. A push to main triggers the pipeline and lands on staging with no manual step.
  • Production is a tag. Cutting a v* git tag is the promotion; there is no separate "deploy to prod" button.
  • Soak first. Let a change sit on staging for roughly 30 minutes before tagging — this is a human judgement, not an enforced gate.
  • Three jobs, chained. test → build-and-push → deploy. A red build never deploys.
  • Rolling deploys. Kubernetes replaces pods one at a time; the gateway readiness probe gates traffic, so a bad image fails the rollout instead of taking the site down.
  • Fastest undo. kubectl rollout undo deploy/gateway, or re-tag the last good commit. Never hand-edit production.

The pipeline: three jobs

infra platform

The jobs are chained with needs, so each only runs if the one before it passed. The same three jobs run for both staging and production — only the target cluster changes.

JobDepends onWhat it does
test—npm ci then npm test across the platform workspaces on Node 20. A red build never deploys.
build-and-pushtestLoops over the services gateway, accounts, billing, usage, notifications; builds one image per service tagged with the commit SHA and pushes to ghcr.io/demo-co/platform-<svc>.
deploybuild-and-pushPoints kubeconfig at eks-<env>, runs kubectl apply -f kubernetes/, then waits for the gateway rollout to be healthy.

The image path is always ghcr.io/demo-co/platform-<svc>:<commit-sha> — one image per service, tagged by the commit that built it. The test job is the deploy half of the merge bar; see Testing.

aws eks update-kubeconfig --name eks-prod
kubectl apply -f kubernetes/
kubectl rollout status deploy/gateway --timeout=120s
What the deploy job waits onThe deploy job waits on the gateway rollout (deploy/gateway, 120s timeout). Internal services are applied by the same kubectl apply, but the sources do not describe a rollout-wait for them — don't assume one.

Rolling deploys and the health gate

infra platform

Kubernetes rolls pods one at a time. The readiness probe gates traffic to each new pod, so a bad image fails the rollout rather than taking the site down. The gateway runs replicas: 3, and its readiness probe is GET /healthz on port 3000.

  • The probe. The gateway implements GET /healthz, returning { ok: true }; the manifest checks it with an initialDelaySeconds of 5.
  • No auth on the probe. /healthz requires no session — that contract is owned by The Gateway & Request Flow.
  • Bad image, contained. If new pods never go ready, traffic stays on the old pods and the rollout stalls instead of erroring out to customers.

To watch a deploy land:

aws eks update-kubeconfig --name eks-prod
kubectl rollout status deploy/gateway
kubectl get pods -l tier=internal
SHA-to-manifest wiring is not documentedThe pipeline pushes SHA-tagged images, while kubernetes/gateway.yaml references a :latest image tag. How kubectl apply resolves the commit SHA into the manifest at deploy time is not shown in the sources — treat the exact mechanism as unknown.

Rollback and the rules

Release & deployment (wiki article 1003) infra

Most incidents follow a deploy, so rolling back is the on-call's first lever — see Observability & On-Call and Incident Response & Postmortems. There are two ways to undo a release.

Fastest path — roll the Deployment back
Run kubectl rollout undo deploy/gateway. This rolls the Deployment back to its previous ReplicaSet — effectively a one-click redeploy of the previous image tag.
Bad release tag — re-tag
Re-tag the last good commit and let the pipeline ship it through the same path it shipped the broken one.
Never hand-edit productionDon't fix production with kubectl edit — the next deploy overwrites it and the change is lost. All production change goes through git.

Migrations in a release

infra

Schema changes are applied by make migrate as part of the release, before the new image takes traffic. Because pods roll one at a time, old pods briefly run against the new schema during the rollout — so every migration must be backward compatible.

The mechanics — expand-then-contract across two deploys — are owned by Database Migrations. This section only states that migrations run as part of the release.
