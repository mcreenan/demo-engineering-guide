Change the schema in two deploys: expand first, then contract. Demo Co ships continuously and Kubernetes rolls pods one at a time, so for a short window old and new code run against the same database — every migration must be safe for that overlap.

In shortExpand then contract over two deploys; each service owns its migrations under platform/services/<name>/migrations; make migrate applies them in filename order; add columns nullable or with a default, backfill in batches, build indexes CONCURRENTLY; never rename or drop a column in the same deploy that stops using it; a failed migration fails the deploy — fix forward.

This section owns migration mechanics. The deploy pipeline and rollout behavior live in Release & Deployment; the column-level schema lives in Core Domain & Data Model. All migration tooling is in .

Why expand/contract

Database migrations (wiki article 1007) runbooks/deploys.md

Migrations are applied as part of the release, before the new image takes traffic. Because pods roll one at a time, old pods keep serving against the new schema during the rollout, so every migration must be backward compatible with the code currently running.

The way to satisfy that is to split a breaking change across two deploys:

Expand (deploy 1)
Add the new shape alongside the old one. The schema now supports both the old and new code, so old pods keep working while the rollout proceeds.
Contract (deploy 2)
Once every pod runs the new code and nothing reads the old shape, remove it.

A worked rename: full_name → display_name

Database migrations (wiki article 1007)

This is the documented pattern, not a committed file — renaming a column directly would break old pods mid-rollout. Split it in two:

  1. Deploy 1 (expand). Add display_name nullable, backfill it from full_name, and ship code that writes both columns and reads the new one. Old pods still read and write full_name, so they keep working.
  2. Deploy 2 (contract). Stop writing full_name, then drop it. By now every pod reads display_name, so nothing depends on the old column.

Where migrations live and how they run

Database migrations (wiki article 1007) services/*/migrations, scripts/migrate.sh, Makefile

Each service owns its own schema and its own migrations under platform/services/<name>/migrations. A service never migrates another service’s schema — that is the schema-per-service boundary from ADR-002 (see Architecture at a Glance and Platform Services).

Files use a numeric prefix and apply in filename order, e.g. 001_init.sql. Today each of the four data-owning services has exactly one migration, its 001_init.sql.

ServiceMigrations
accountsservices/accounts/migrations
billingservices/billing/migrations
usageservices/usage/migrations
notificationsservices/notifications/migrations
gatewaynone — it owns no schema, so it is not migrated

make migrate runs ./scripts/migrate.sh once per service in a fixed order — accounts, billing, usage, notifications. It requires DATABASE_URL, and applies every SQL file in a directory in filename order:

# scripts/migrate.sh, per file, in filename order
set -euo pipefail
psql "$DATABASE_URL" -v ON_ERROR_STOP=1 -f "$file"

Because the script runs under set -euo pipefail with ON_ERROR_STOP=1, a single SQL error aborts the whole run — which is how a bad migration fails the deploy rather than half-applying.

Model the existing styleThe committed 001_init.sql files show the house syntax to follow: UUID primary keys with DEFAULT gen_random_uuid(), timestamptz DEFAULT now(), role checks like CHECK (role IN ('owner','admin','member')), and partial unique indexes such as one owner per org. When a migration fails

Database migrations (wiki article 1007) runbooks/deploys.md

A failed migration fails the deploy, and the old version keeps serving — the rollout stops rather than taking the site down. Fix forward with a new migration; do not roll the migration back.

Fix forward, don’t roll backWrite a new migration that corrects the problem and let the pipeline apply it. Rolling a migration back risks leaving the schema and the running code out of step — the same reason expand and contract are split across two deploys.
