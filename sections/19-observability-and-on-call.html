On-call keeps production healthy and is the first responder when it is not. The primary on-call is paged when the gateway 5xx rate exceeds 2% for 5 minutes, and escalates to the service owner if the incident is not resolved within 15 minutes.

  • Trigger. Gateway 5xx rate > 2% for 5 minutes pages the primary on-call.
  • Escalate. Hand off to the service owner if it is not resolved within 15 minutes.
  • Look at the gateway first. It is the single ingress, so most symptoms show up there.
  • Check recent change. Most incidents follow a deploy — roll it back before deeper debugging.

Paging & escalation

The on-call signal is the gateway's error rate, because the gateway is the single public entrypoint for all traffic. When errors cross the threshold the primary on-call is paged; if the page is not resolved in time it escalates to the owner of the affected service.

Triage runbook

On-call runbook (wiki article 1002) infra

Start at the gateway. It is the single ingress, so its dashboard and logs are the fastest way to see which route is failing and whether the problem is one service or the whole edge.

  1. Check the gateway dashboard. Find the failing route to narrow the blast radius.
  2. Restart a down service. If a single service is down, restart it with kubectl rollout restart deploy/<svc>.
  3. Shed read load. If Postgres is saturated, shed read load to the replica (see the note below).

Reach for the gateway's pods and logs first. The on-call runbook uses these two commands:

# Look at the gateway first
kubectl get pods -l tier=edge
kubectl logs -l app=gateway --tail=100

A restart or rollback will not black-hole traffic: the gateway's readiness probe is /healthz (no auth), and Kubernetes rolls pods one at a time, so a bad image fails the rollout rather than taking the site down.

The tier=edge selector is the runbook's command, not a pod labelIn the gateway manifest the tier: edge label is on the Deployment metadata, while the pod template carries only app: gateway. The runbook lists kubectl get pods -l tier=edge verbatim, so use it as written but fall back to app=gateway if it returns nothing.

Common causes & fixes

On-call runbook (wiki article 1002) infra

Most incidents fall into a few buckets. Match the symptom to a cause, apply the fix, and escalate if it does not resolve within 15 minutes.

CauseSymptomFix
Bad deployErrors started right after a deploy or while one is in flightRoll back: kubectl rollout undo — resolves most Sev 1/2s
Database saturationA slow query or connection leak in one serviceCheck rds-prod CPU and connections in CloudWatch; find the offending service by its schema in pg_stat_activity
Expired / rotated secretA service crash-looping on startupInspect with kubectl describe pod for a missing or rotated secret

Postgres schema names map to services, so pg_stat_activity tells you who is generating the load. The schema names are the service names: accounts, billing, usage, or notifications. The services themselves live in ; the runbooks and Kubernetes manifests live in .

Roll back first, debug secondThe full rollback mechanics — kubectl rollout undo deploy/gateway and re-tagging the last good commit — are owned by Release & Deployment. When a deploy is the suspect, roll it back before deeper investigation.
"Shed read load to the replica" is a runbook step, not a confirmed targetThe on-call runbook lists shedding read load to a replica as a remediation, but no read replica is defined in the available Terraform, and the mechanism (how to shed load, the replica endpoint) is not documented in the sources. Treat it as the runbook's instruction and confirm the current setup before relying on it.
