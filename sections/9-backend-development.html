The backend is a Node + TypeScript monorepo in , where each service is a class — Gateway, AccountsService, BillingService, UsageService, and NotificationsService — and every service method is scoped to an org passed in by the gateway, never read from a request body.

  • Stack. Node >= 20, TypeScript, express, pg, and zod, organized as npm workspaces under services/*.
  • One iron rule. Methods take orgId as the first argument; downstream services trust the gateway's x-org-id header and ignore any orgId in the body.
  • Internal addressing. Services resolve at <name>.internal inside the cluster and listen on container port 8080; the gateway listens on 3000.
  • Tooling. typecheck is tsc --build, tests are vitest, lint is eslint — all driven by the Makefile.

The service-as-class pattern

platform/services/gateway/app.ts platform/services/accounts/index.ts platform/docs/architecture.md

Each service is implemented as a single class. The Gateway is the only publicly-exposed service: it authenticates every request, then routes to an internal service. Internal services are never reachable directly from outside the cluster and never call each other directly — a cross-service read comes back out through the gateway.

Org-scoping: never trust a body orgId

platform/services/gateway/proxy.ts platform/services/accounts/index.ts platform/docs/architecture.md

This is the central backend convention. Every service method is scoped to an org, and the only trusted source of that org is the gateway. After the gateway verifies the session, its proxy attaches two identity headers from that session — x-user-id and x-org-id — and downstream services scope every query to the verified org.

  • Take orgId first. Method signatures lead with orgId so scoping is unmissable.
  • Trust the header, not the body. Scope queries to x-org-id; never read an orgId from the request body.
  • It is a security invariant. A missing org scope is a security bug — see Security & Secrets.
Never trust an orgId from a request bodyThe verified org comes from the session, relayed as x-org-id. Reading an org identifier the client could set would let one tenant read another's data.

sequenceDiagram

The proxy builds the base URL from the service name with no explicit port in code, copies the method, sets the JSON content type plus the two identity headers, sends the body as JSON for non-GET requests, and relays the upstream status and text. The container port 8080 comes from the Kubernetes manifests in , not from the proxy code.

Append-only usage events

platform/services/usage/index.ts platform/services/billing/index.ts

The usage_events table is append-only: record(orgId, units) inserts a row, and events are never updated or deleted. totalForPeriod(orgId, start, end) sums units across a half-open time window, and that total is what billing reads.

  • Write. record appends an event with occurred_at = now(); corrections are new events, not edits.
  • Read. totalForPeriod returns the summed units for an org over the requested window.
  • Billing reads through the gateway. When finalizing an invoice, BillingService fetches the usage total from the usage service via the gateway (GET /usage/total?period=current) — it never queries the usage schema directly.

Overage math and plan/unit pricing belong to Billing, Plans & Usage Metering.

Notifications: enqueue + delivery worker

platform/services/notifications/index.ts

Notifications split into two halves. enqueue(orgId, template, channel) inserts a pending row and returns immediately; a background worker tick, deliverPending(), claims and sends pending rows, marking each sent or failed with retry and backoff.
